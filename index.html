Churn prediction - Kaggle competition¶

Go to project Github repository¶

¶

This a an exploratory analysis and prediction with XGBoost for Kaggle competition:

"WSDM - KKBox's Churn Prediction Challenge Can you predict when subscribers will churn?"

The competition is over but you can still download the data, read notebooks of participants and learn!
And then submit your prediction to see how well your model perform.

Overview of the task¶

The goal of the competition is to predict if the user of the music streaming service will renew the subscription. Usually the subscription lasts 30 days and can be automatically renewed.
Churn definition (in data description on Kaggle)

The criteria of "churn" is no new valid service subscription within 30 days after the current membership expires.

Summary of my best solution¶

This solution scored 0.11637 on Public Leaderboard what gives it 41th place (best 8%)


I've used PySpark to filter user_logs.csv (file is too big to lead into memory).

Best model features:

  • last_is_auto_renew: (boolean) if last subscription will be renewed automatically,
  • cancel_sum_4weeks: number of cancelation during the last 4 weeks,
  • days_from_start: number of days from the first transaction,
  • last_payment_plan_days: length of the current subscription in days,
  • last_month_logs_num: number of active days during last 4 weeks,
  • num_100_sum: number of transactions during the last transaction day.


Algorithm and parameters:

  • XGBoost
  • max_depth=3
  • scale_pos_weight=0.8
  • learning_rate=0.1

Trasactional Features¶

The challenge winner, Bryian Gregory, gives a few tips in his article,
"Predicting Customer Churn: Extreme Gradient Boosting with Temporal Data"
https://medium.com/@bryan.gregory1/predicting-customer-churn-extreme-gradient-boosting-with-temporal-data-332c0d9f32bf

He gives a hint that the features should be relative to a cutoff day. He calls it "Relative Refactoring Method".
So we should transform the day of first transaction into the number of days from the first transaction.
Only if days are "special" like holidays we could live them as exact days. We should not mix those approaches.
Here, I'll calculate a few relative features, e.g.:

  • days_from_start = Number of days from first transaction

    Full list of features below in the section 'Modeling with XGboost'.

Logs features¶

Logs file is too big to load it into the computer memory. Let's use pyspark to filter out monthly logs!

I've used this code to filter logs by month:

from pyspark import SparkContext from pyspark.sql import SparkSession sc = SparkContext("local", "First App") my_spark = SparkSession.builder.getOrCreate()

file_path = user_logs_path user_logs = my_spark.read.csv(file_path, header=True) user_logs.show() user_logs.createOrReplaceTempView("user_logs") print(my_spark.catalog.listTables())

query_january = "SELECT* FROM user_logs WHERE date >= 20170101 AND date <= 20170131" january_logs = my_spark.sql(query_january) january_logs.coalesce(1).write.format('csv').save("january_logs", header='true')

query_february = "SELECT* FROM user_logs WHERE date >= 20170201 AND date <= 20170228" february_logs = my_spark.sql(query_february) february_logs.coalesce(1).write.format('csv').save("february_logs", header='true') my_spark.stop()

Features importance¶

  • We need to remember that there are many types of feature importance: 'weight', 'gain', 'cover',
  • Default: weight: the number of times a feature is used to split the data across all trees. It may be lower for features with lower cardinality (like True/False features, so our "last_is_auto_renew"),
  • gain: the average gain across all splits the feature is used in.
